Relatório MCP/Agents

Situação atual: o modelo MCP + Capability Marketplace está integrado em boa parte. O agente consegue descobrir capabilities, ver contratos, comprar acesso com USDT, executar com metering/quota, criar webhooks e criar intent M2M para pagamento PIX. O produto “agente usa nosso rail para pagar PIX/cartão” existe no código, mas PIX está mais completo que cartão.

Modelo de Cobrança

Marketplace/capability:

O agente compra plano em USDT na BSC.
take_rate_bps = 2000, ou seja, 20%.
Exemplo: plano de 100 USDT gera 20 USDT ChainFX e 80 USDT provider.
Depois da compra, o agente recebe access grant com quota.
Execução consome quota, não cobra taxa extra por chamada hoje.
M2M PIX:

M2M_PIX_FEE_BPS=1000 por default, ou seja, 10%.
O agente informa valor BRL final.
Sistema calcula:
gross_usdt = amount_brl / USDTBRL
fee_usdt = gross_usdt * 10%
required_usdt = gross_usdt + fee_usdt
Agente deposita USDT no TREASURY_HOT.
Onchain worker detecta depósito.
Settlement worker paga PIX via Efí.
Daily cap default: M2M_MAX_DAILY_OUTFLOW_BRL=50000.
M2M cartão:

Existe type=credit_card e fee default M2M_CREDIT_FEE_BPS=1900, ou seja, 19%.
Mas auditoria técnica: cartão ainda não está realmente completo como rail de liquidação.
O worker atual liquida via Efí PIX e exige pix_key.
Não existe coleta/tokenização de dados de cartão, boleto/fatura, beneficiário de cartão, nem adapter real de cartão.
Então hoje “cartão” está modelado na API/config/taxa, mas não está pronto como produto operacional.
Como o Agente Acessa Hoje

Fluxo normal:

Cliente/developer chama POST /agent/connect.
Sistema cria identidade do agente e policy.
Agente usa MCP autenticado via API key/Bearer.
Agente lista capabilities.
Agente compra plano via purchaseCapability ou HTTP marketplace.
Agente paga on-chain em USDT.
Compra vira grant/access token.
Agente executa capability com executeCapability.
Sistema debita quota e registra execution event.
Fluxo M2M PIX:

Agente chama createPaymentIntent no MCP ou POST /agent/v1/pay.
Informa type=pix, amount_brl, pix_key, agent_wallet, idempotency_key.
Sistema aplica policy do agente.
Sistema retorna required_usdt, payment_address, expires_at.
Agente deposita USDT.
Onchain worker confirma.
M2M settlement paga PIX via Efí.
Agente consulta com getPaymentIntent.

Gaps Críticos

Cartão de crédito não está completo. Precisa adapter real, campos próprios, antifraude e settlement específico.
payments_fx ainda é mais capability/contrato do que execução financeira real dentro de executeCapability; o M2M real está em createPaymentIntent.
Public MCP registry ainda não está publicado/listado.
Enterprise routing policies existem no banco, mas ainda precisam UI/admin e regras comerciais mais refinadas por cliente.
Webhook UI existe, mas precisa maturar eventos finais de M2M: hoje há nomes divergentes como m2m.settlement.done vs lista UI com m2m.settled.
Falta painel operacional de risco: intents pendentes, depósitos sem match, outflow diário, falhas Efí, replay/idempotência.
Falta precificação dinâmica por agente/plano/volume. Hoje M2M usa env global.
